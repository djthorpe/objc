1 # objc runtime
2 
3 This is a minimal Objective C runtime written in C, designed to be portable across different platforms, including ARM and x86 architectures, mostly targeting embedded systems. It uses the "gcc" ABI for the moment, as that is the most portable across different platforms.
4

6 - [View the list of all files](https://djthorpe.github.io/objc/files.html)

9 ## Requirements
10 
11 You will minimally need the following tools to build the runtime:
12 
13 - **Build system** `make` and `cmake` - for the build system
14 - **Compiler** `clang` or `gcc` - for compiling the runtime (`clang` is not supported on Apple Silicon). You can use the environment variable `CC` to specify the compiler, e.g. `CC=clang` or `CC=gcc`.
15 - **Library Dependencies** `openssl` for Linux and Darwin - for the hash functions, which are used in the runtime and Foundation framework.
16 - **Documentation** `docker` is needed for generating the documentation from the source code.
17 - **Cross-Compilation** For cross-compilation for embedded systems based on some ARM variant, get the ARM LLVM toolchain: <https://github.com/ARM-software/LLVM-embedded-toolchain-for-Arm/releases>. Install this to the `/opt` directory. You can use the environment variable `TOOLCHAIN_PATH` to specify the path to the toolchain, e.g. `TOOLCHAIN_PATH=/opt/LLVM-ET-Arm-19.1.5-Darwin-universal`.
18 
19 ## Building the libraries
20 
21 Several static libraries are currently built:
22 
23 - `objc-gcc` - the Objective C runtime library using the ancient GCC ABI
24 - `runtime-sys` - a minimal set of system functions, needed to bind the runtime to the underlying system, on a per-platform basis. Includes cryptographic hash functions (MD5, SHA-256).
25 - `runtime-hw` - hardware interfaces like SPI, I2C, GPIO. These are stubs for Linux and Darwin, but I guess some of these interfaces couple be implemented later.
26 - `drivers` and `runtime-pix` have a long way to go but should provide hardware driver and display support eventually.
27 - `Foundation` - a minimal set of classes, to support memory management and basic types such as string, array, dictionary, date, time and number.
28 - `Application` - the application framework, which provides a runloop and handles input, display, power and sensors.
29 
30 A lot of this code is in development still. See the "examples" or documentation on what's been implemented so far.
31 
32 Download the source code from GitHub:
33 
34 ```bash
35 git clone git@github.com:djthorpe/objc.git
36 cd objc
37 ```
38 
39 The `TOOLCHAIN_PATH` environment variable should point to the directory where the toolchain is installed.
40 For Macintosh, you can use Homebrew to install the GCC toolchain, which is required for compiling the runtime on MacOS:
41 
42 ```bash
43 # Compile with GCC 15 for MacOS
44 brew install gcc@15
45 TOOLCHAIN_PATH=/opt/homebrew RELEASE=1 CC=gcc-15 make
46 ```
47 
48 Once you've made the libraries, use the `make tests` target to run the unit tests. There is information about test coverage in the [tests directory](src/tests/README.md).
49 
50 ### Running Tests
51 
52 On macOS, build and run all tests:
53 
54 ```bash
55 TOOLCHAIN_PATH=/opt/homebrew CC=gcc-15 RELEASE=1 make tests
56 ```
57 
58 Run an individual test target (replace `<test_target>` with e.g. `fs_09`):
59 
60 ```bash
61 cmake --build build --target <test_target>
62 ctest --output-on-failure --test-dir build --tests-regex <test_target>
63 ```
64 
65 See the [tests README](src/tests/README.md) for a catalog of tests (fs_01..fs_09 and others) and details.
66 
67 You can target different architectures by setting the `TARGET` environment variable. For a RP2040-based board, you can use the `clang` compiler with the ARM toolchain. The `TARGET` environment variable should be set to the target architecture, such as `armv6m-none-eabi` for the RP2040 Pico board:
68 
69 ```bash
70 # Compile for the RP2040 Pico board
71 CC=clang TARGET=armv6m-none-eabi TOOLCHAIN_PATH=/opt/LLVM-ET-Arm-19.1.5-Darwin-universal RELEASE=1 make
72 ```
73 
74 See the list of supported targets in the [cmake](https://github.com/djthorpe/objc/tree/main/cmake) directory.
75 You can exclude the environment variable `RELEASE=1` to build debugging versions of the libraries.
76 
77 For Pico cross-compilation, after building, you can load a specific test onto a Pico W board (example for `<test_target>`):
78 
79 ```bash
80 cmake --build build --target <test_target> && picotool load -x build/src/tests/<test_target>/<test_target>.uf2
81 ```
82 
83 For Linux cross-compilation using Docker (Bookworm):
84 
85 ```bash
86 docker run --rm -i -v $(pwd):/root bookworm-builder bash -c "cd /root && make clean && CC=gcc make"
87 docker run --rm -i -v $(pwd):/root bookworm-builder bash -c "ctest --output-on-failure --test-dir build --tests-regex <test_target>"
88 ```

108 ## Documentation
109 
110 To build the API documentation, you will need to have `docker` installed. The documentation is built using `doxygen` through `docker` so you don't need to have it installed directly.
111 
112 ```bash
113 make docs
114 open docs/index.html
115 ```
